# Commitlint Configuration Design

## Context

This change introduces commit message linting for the current monorepo, aligned with `.cursor/rules/git-commit-convention.mdc`.
The scope is intentionally minimal:

- Add `commitlint.config.cjs`
- Enforce checks at Git commit time via Husky `commit-msg`
- Keep existing `pre-commit` flow unchanged

No additional contract or mirror documentation is included in this change.

## Goals

- Enforce Conventional Commits header shape: `type(scope): subject`
- Restrict allowed `type` values to the agreed list
- Ensure non-empty `subject`
- Forbid CJK characters in any non-empty `subject`, `body`, or `footer`
- Preserve explicit ignore behavior for merge and revert auto-generated commits

## Non-goals

- No repository-wide refactor of existing commits
- No CI-only enforcement (local guard is required)
- No changes to lint-staged / Biome rules
- No additional docs synchronization in this task

## Architecture And Boundaries

The design keeps responsibilities isolated per file:

- `commitlint.config.cjs`
  - Owns all linting policy (types, language constraints, ignores)
  - Exports static configuration only
- `.husky/commit-msg`
  - Thin execution layer only
  - Runs `pnpm exec commitlint --edit "$1"`
- `package.json`
  - Tool wiring only (`@commitlint/cli` and config package dependencies)
  - No policy logic

This separation keeps hook execution stable while allowing future policy updates in one place.

## Rule Design

### Base Conventional Rules

- `type-enum` allows only:
  - `feat`, `fix`, `docs`, `style`, `refactor`, `perf`, `test`, `build`, `ci`, `chore`
- `type-empty`: disallow
- `subject-empty`: disallow
- `subject-full-stop`: disallow trailing `.`

### Language Constraint

Add a custom plugin rule that rejects CJK ideographs in non-empty message parts:

- `subject`
- `body`
- `footer`

Regex basis: `[\u4E00-\u9FFF]`

Rationale:

- This directly enforces the project rule that all non-empty commit text in Git metadata is English
- `subject/body/footer` are checked uniformly so there is no bypass via multiline message parts

### Ignore Strategy

Use `ignores` in commitlint config for known exempt patterns:

- `Merge branch '…'`
- `Merge remote-tracking branch '…'`
- `Merge pull request #…`
- `Revert "…"` / `Revert '…'`

Rationale:

- These are generated by Git/GitHub tooling and are intentionally exempt in team conventions
- Exemptions are centralized and explicit

## Data Flow

1. Developer runs `git commit`
2. Existing `.husky/pre-commit` runs (lint-staged)
3. Git invokes `.husky/commit-msg` with commit message file path
4. Hook runs `pnpm exec commitlint --edit "$1"`
5. Commitlint loads root `commitlint.config.cjs`
6. If any rule fails, commit exits with non-zero status and is blocked

## Error Handling And Developer Feedback

- Keep default commitlint output for clear rule-level diagnostics
- Use readable custom rule names/messages for fast correction
- Avoid wrapper scripts that hide original error context

## Test Plan

### Should Pass

- `feat(api): add commit message guard`
- `fix: handle null input in parser`
- Multiline message with English-only body and footer

### Should Fail

- Invalid type: `feature: add x`
- CJK in subject: `feat: 增加校验`
- CJK in body/footer
- Trailing full stop in subject: `fix: handle parser.`

### Should Be Ignored

- `Merge pull request #123 from org/branch`
- `Revert "feat(api): add guard"`

### Integration Check

- Confirm `git commit` triggers `.husky/commit-msg`
- Confirm failed message blocks commit
- Confirm success path does not change existing `pre-commit` behavior
